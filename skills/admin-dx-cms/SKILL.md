## ワークフロー

### STEP 1｜相談内容の構造化

```
【依頼者】エンド / 代理店 / 自社　・業種
【目的】何を作りたいか（サイトリニューアル / 新規プラットフォーム / 統合 等）
【現状】現行CMSや技術スタック（WordPress等）
【課題】何が問題か（運用コスト・スケーラビリティ・デザイン自由度 等）
【要件】必須機能（検索・会員・API連携・多言語 等）
【制約】予算・納期・リソース
```

### STEP 2｜プロダクト適性確認（Kurocoを選ぶ理由を整理）

| シグナル | Kuroco推奨理由 |
|---------|--------------|
| 複数サイト管理 | マルチドメイン・マルチテナント標準対応 |
| 外部API連携が多い | REST/GraphQL/Webhook標準装備 |
| デザイン自由度が必要 | ヘッドレス構成でフロント完全自由 |
| 大量アクセス対応 | CDNキャッシュ×APIで耐障害性高 |
| 現場の更新負荷削減 | ノーコード管理画面で非エンジニアでも更新可 |
| セキュリティ懸念 | SaaS自動アップデートでWordPressのような脆弱性管理不要 |

### STEP 3｜プラン提示（3段階）

| プラン | 内容 | 推奨シーン |
|--------|------|----------|
| 標準プラン | Kuroco標準機能＋シンプルなフロント実装 | 予算重視・スピード優先 |
| 推奨プラン（拡張型） | カスタム開発込み・API連携・会員機能 | コンペ推奨・フル要件対応 |
| MVP | 最小機能で早期稼働・段階的拡張 | フェーズ提案・予算未確定時 |

### STEP 4｜カスタム開発箇所の整理（F/B番号付与）

Admin-DX CMS案件の典型的なカスタム開発パターン：

| ID | 区分 | 典型機能 |
|----|------|---------|
| F-01 | フロント | 検索・絞り込みUI（条件フィルタ・ソート） |
| F-02 | フロント | 会員機能・マイページ・お気に入り |
| F-03 | フロント | お問い合わせフォーム・確認・完了ステップ |
| F-04 | フロント | マップ連携（Google Maps等） |
| B-01 | バック | 検索API拡張（複合条件・全文検索） |
| B-02 | バック | 外部CRM/MA連携（Webhook送信） |
| B-03 | バック | 外部認証連携（SSO・LINE Login等） |
| OPS-01 | 設定作業 | Kurocoコンテンツ型定義・ワークフロー設定 |

### STEP 5｜generate_pitch.js の生成

`C:\Users\kamat\scripts\_tpl_generate_pitch.js` をコピーして案件フォルダに配置。
`PITCH_CONFIG` を案件情報で埋めて `node generate_pitch.js` で生成。

出力ファイル命名規則：
```
generated/pitch_{案件スラッグ}_{yyyymm}.pptx
```

## Admin-DX CMS 特有のスライド内容ガイド

### P2 クライアント要件 - 左カラムのブロック推奨構成

```
ブロック1：CMSコンテンツ管理の仕組み（Kuroco API→フロント配信フロー）
ブロック2：主要機能（検索・会員・問い合わせ等、案件固有要件）
```

### P4 技術的な合意事項 - 3カラム推奨構成

```
カラム1（オレンジ）：Kuroco CMS アーキテクチャ（CMS→API→CDN）
カラム2（ブルー）  ：フロントエンド構成（Next.js/Nuxt.js・実装方針）
カラム3（グリーン）：管理・運用フロー（登録→公開→問い合わせ管理）
```

### P6 ユーザーフロー推奨構成

上段（ユーザー行動）：サイト訪問 → 機能利用 → 問い合わせ/コンバージョン → 完了
下段（システム）    ：Kuroco CMS → API配信 → 担当者通知・管理
