# 案件経緯・背景・提案方針サマリー
## 公共交通利用促進キャンペーン｜ビザビ×WWS

---

## 1. 案件の背景

### 代理店（ビザビ）への相談内容
行政（地方自治体）が主体となる**公共交通利用促進プロモーション**の企画において、
GPSスタンプラリー・レシートキャンペーンの仕組みを活用したシステム構築の相談が
代理店経由でビザビに持ち込まれた。

### クライアントが解決したい課題

| 課題 | 内容 |
|------|------|
| 交通事業者の負担ゼロ | 車内へのQRコード設置・シール掲出は一切不可 |
| 不正参加の防止 | 車での移動による不正チェックイン排除 |
| アプリ不要 | 専用アプリDL不要・LINEログインのブラウザ参加 |
| 多交通手段対応 | バス・電車・路面電車・フェリー・タクシー全対応 |

### 対象エリア・交通機関
- **固定路線**：バス・電車・路面電車・フェリー
- **自由路線**：タクシー（GPS固定スポット設置不可）

---

## 2. クライアントの要件（原文ベース）

### 判定ロジック①｜固定路線（GPS）
> GTFSオープンデータの緯度経度情報をシステムに一括登録し、
> 「乗車地（出発）」と「降車地（到着）」の2箇所でのGPSチェックインをもって
> 1乗車と判定。インターバル制限等で車移動での不正を防ぎたい。

### 判定ロジック②｜自由路線（タクシー）
> GPSスポットの設定ができないため、降車時の
> 「レシート画像アップロード（OCR解析・重複チェック）」をもって1乗車と判定。

### その他要件
- ポイント蓄積→賞品応募（または即時抽選）まで一気通貫
- 個人情報のセキュアな取得
- 類似要件のデモ画面提示を求められている
- GTFSデータの一括取り込み可否を確認したい
- コンペ参加案件のため**競合他社との差別化**が必要

---

## 3. ビザビとしての提案方針

### 基本スタンス
「要件を100%満たせなくても、**目的（公共交通利用促進）を達成できる提案**をする」

### プロダクト選定の結論

```
SmartStampCard（スタンダード）
  └── GPS乗降2点チェックイン → 固定路線の乗車判定
  └── マイレージ機能（OP）  → ポイント累積・賞品応募
  └── LINEログイン（OP）    → ブラウザ参加・認証基盤

SmartRecieco（WEB版 マイレージ＋LINEログインOP）
  └── OCR自動審査           → タクシー領収書の乗車判定
  └── 事務局日次審査        → 重複・不正チェック

【連携】Receico承認 → Cardポイント加算（カスタム開発）
```

### 提案の3段階構成

| プラン | 内容 | 費用感 |
|--------|------|--------|
| **標準プラン** | 既存機能のみ・手動スポット登録・最短10日納期 | 低 |
| **推奨プラン（カスタマイズ型）** | GPS乗降ペア判定＋Receico連携＋SSO | 中〜高 |
| **MVP（最小構成）** | StampCardのみ・タクシーは降車GPSで代替 | 低・最速 |

### コンペ戦略
- **フェーズ提案**で行政の予算・スケジュール感に合わせる
  - フェーズ1：MVP（早期稼働・実績づくり）
  - フェーズ2：GTFS連携・完全版
- 行政案件は「**スケジュール確実性**」と「**費用の透明性**」が刺さる

## 4. 技術的な合意事項（打合せで決定した設計）

### 乗降2点チェックインの仕組み
- 管理画面に各駅・停留所を事前登録（GTFS等から取り込み）
- ユーザーが「乗車」→「降車」の順でGPSスタンプを2回押す
- 乗車地＋降車地のペア成立で1ポイント付与
- どちらか片方のみでは付与しない（B-01 カスタム開発）

### タクシーレシートの運用フロー
- ユーザー：領収書を撮影・アップロード（写真を選択ボタン）
- OCR自動チェック：タクシー会社名・日時をタグ条件で自動判定
- **OCR結果はユーザー画面に表示しない**（管理側のみ）
- 事務局が1日1回目視確認（最大100件/日）→ 承認でCardにポイント反映
- ユーザーへは「2〜3営業日以内に付与」と表示

### ポイント統合の設計
- StampCardを**ポイント管理の基盤**とする
- ReceicoはLINEトークン引き継ぎ（SSO）でCardからシームレス遷移
- タクシー承認ポイントはReceico→CardへのAPI連携（Webhook方式）で反映
- Cardのマイページに「GPS乗車分」「タクシー承認済」「審査中（予定）」を内訳表示

## 5. カスタム開発が必要な箇所（SE算定対象）

| ID | 機能 | 難易度 |
|----|------|--------|
| F-01 | マイページ：乗車種別ボタン追加 | 小 |
| F-02 | GPS STEP UI：乗降ラベル変更 | 小 |
| B-01 | 乗降2点ペア判定ロジック | **大** |
| B-02 | Receico→Cardポイント連携 | **中〜大** |
| F-03/B-03 | Card→Receico SSO連携 | 中 |
| F-04 | 審査ステータス表示 | 小〜中 |
| F-05 | 応募モーダル：ポイント内訳表示 | 小 |
| B-04 | OCRタグ審査条件設定（設定作業のみ） | 設定のみ |

### WWS社内確認が必要な事項（見積前）
1. Cardのスポットにペア属性・セッション管理が標準対応しているか
2. ReceicoにWebhookが存在するか（B-02の実装方式の分岐点）
3. 両プロダクトのLINEチャネルが共通か（B-03の実装方式の分岐点）
4. CardマイページのHTMLカスタマイズ可能範囲（F-04/F-05に影響）
