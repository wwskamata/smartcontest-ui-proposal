1. 案件の背景

代理店（ビザビ）への相談内容

行政（地方自治体）が主体となる公共交通利用促進プロモーションの企画において、GPSスタンプラリー・レシートキャンペーンの仕組みを活用したシステム構築の相談が代理店経由でビザビに持ち込まれた。

クライアントが解決したい課題

課題内容
交通事業者の負担ゼロ車内へのQRコード設置・シール掲出は一切不可
不正参加の防止車での移動による不正チェックイン排除
アプリ不要専用アプリDL不要・LINEログインのブラウザ参加
多交通手段対応バス・電車・路面電車・フェリー・タクシー全対応

対象エリア・交通機関

固定路線：バス・電車・路面電車・フェリー（GPS判定）
自由路線：タクシー（GPSスポット固定設置不可 → レシートOCR判定）

2. クライアントの要件

判定ロジック① ｜ 固定路線（GPS）

GTFSオープンデータの緯度経度情報をシステムに一括登録し、「乗車地（出発）」と「降車地（到着）」の2箇所でのGPSチェックインをもって1乗車と判定。インターバル制限等で車移動での不正を防ぎたい。

判定ロジック② ｜ 自由路線（タクシー）

GPSスポットの設定ができないため、降車時の「レシート画像アップロード（OCR解析・重複チェック）」をもって1乗車と判定。

その他要件

✅ポイント蓄積→賞品応募（または即時抽選）まで一気通貫
✅個人情報のセキュアな取得
✅類似要件のデモ画面提示が求められている
✅GTFSデータの一括取り込み可否を確認したい
⚡コンペ参加案件のため競合他社との差別化が必要
3. 提案方針

基本スタンス

「要件を100%満たせなくても、目的（公共交通利用促進）を達成できる提案をする」

プロダクト選定の結論

SmartStampCard（スタンダード）
GPS乗降2点チェックイン → 固定路線の乗車判定
マイレージ機能（OP） → ポイント累積・賞品応募
LINEログイン（OP） → ブラウザ参加・認証基盤
SmartRecieco（WEB版 マイレージ＋LINEログインOP）
OCR自動審査 → タクシー領収書の乗車判定
事務局日次審査 → 重複・不正チェック
🔗 連携：Receico承認 → Cardポイント加算（カスタム開発・B-02）

提案の3段階構成

標準プラン

既存機能のみ・手動スポット登録・最短10日納期

費用感：低
コンペ推奨
推奨プラン（カスタマイズ型）

GPS乗降ペア判定＋Receico連携＋SSO

費用感：中〜高
MVP（最小構成）

StampCardのみ・タクシーは降車GPSで代替

費用感：低・最速

コンペ戦略

行政案件は「スケジュール確実性」と「費用の透明性」が刺さる。フェーズ提案で予算・スケジュール感に合わせる。

フェーズ内容狙い
フェーズ1MVP（早期稼働）実績づくり・競合排除
フェーズ2GTFS連携・完全版継続受注・拡張
4. 技術的な合意事項

乗降2点チェックインの仕組み

手順内容
①管理画面に各駅・停留所を事前登録（GTFS等から取り込み）
②ユーザーが「乗車」→「降車」の順でGPSスタンプを2回押す
③乗車地＋降車地のペア成立で1ポイント付与
⚠️どちらか片方のみでは付与しない（B-01 カスタム開発）

タクシーレシートの運用フロー

ステップ内容
ユーザー領収書を撮影・アップロード（写真を選択ボタン）
OCR自動タクシー会社名・日時をタグ条件で自動判定
⚠️ 重要OCR結果はユーザー画面に表示しない（管理側のみ）
事務局1日1回目視確認（最大100件/日）→ 承認でCardにポイント反映
ユーザー表示「2〜3営業日以内に付与」と表示

ポイント統合の設計

基盤StampCardをポイント管理の基盤とする
SSOReceicoはLINEトークン引き継ぎ（SSO）でCardからシームレス遷移
連携タクシー承認ポイントはReceico→CardへのAPI連携（Webhook方式）で反映
表示「GPS乗車分」「タクシー承認済」「審査中（予定）」を内訳表示

ID 区分 機能名 要件概要 難易度 確認事項
F-01 フロント マイページ
ボタン追加		 標準の汎用スタンプボタンを2種に分岐。①「乗車スタンプを押す（GPS）」②「タクシーレシートを登録する」（Receico SSO遷移） 小 —
F-02 フロント GPS STEP UI
ラベル変更		 STEPインジケーターのラベルを「乗車地スタンプ→降車地スタンプ→完了」に変更。スポット確認カードに「乗車地：／降車地：」プレフィックスを付与 小 文言変更がCMS設定で可能か確認
B-01 バック 乗降2点
ペア判定ロジック		 スポットにdeparture／arrivalフラグを付与。セッション内で「乗車地取得済み」を保持し、降車地スタンプはペア成立時のみ1pt付与。片方のみ取得ではポイント付与しない 大 スポットへのフラグ付与・セッション管理がCard標準で対応可能か確認
B-02 バック Receico→Card
ポイント連携		 Receico事務局承認を契機にCardへ1pt付与。Webhook方式が理想。Webhook未対応の場合は日次バッチ（承認済み差分取得→Card API）で代替。失敗時リトライ3回＋エラーログ 中〜大 Receicoに承認Webhookが存在するか確認（実装方式の分岐点）
F-03
B-03		 フロント バック Card→Receico
SSO連携		 CardマイページのタクシーボタンからRecieco画面へ遷移時、LINEトークンを引き継ぎ再ログイン不要にする。チャネル共通の場合はトークン渡し、別チャネルの場合はワンタイムトークン方式 中 両プロダクトのLINEログインチャネルが共通か確認（実装方式の分岐点）
F-04 フロント 審査ステータス
表示		 マイページ履歴に「審査中（オレンジ）」「承認済（グリーン）」バッジを表示。審査中ポイントは合計に含めず「+N pt 予定」として別表示 小〜中 CardのマイページHTMLカスタマイズ範囲を確認
F-05 フロント 応募モーダル
ポイント内訳表示		 応募確認モーダル内に内訳を表示（🚃乗車GPS分／🧾タクシー承認済／⏳審査中予定）。審査中分はグレーアウトで応募不可 小 —
B-04 設定作業 OCRタグ
審査条件設定		 Receico管理画面でタクシー会社名リスト・期間条件・重複チェックを設定。不正チェックバッチを1日1回実行に設定。コーディング不要・設定作業のみ 設定のみ 対象タクシー事業者リストをクライアントから入手

大 新規ロジック実装が必要
