## 3. 提案方針

### プロダクト構成（確定）

| プロダクト | 用途 |
|-----------|------|
| Admin-DX CMS（Kuroco） | コンテンツ管理・API配信・会員管理・問い合わせワークフロー |
| Next.js（フロントエンド） | 物件検索UI・一覧・詳細・フォーム・マイページ |

### 3段階プラン

| プラン | 内容 | 費用感 |
|--------|------|--------|
| 標準プラン | Kuroco標準機能＋シンプル一覧・詳細・問い合わせ | 低 |
| **推奨プラン（拡張型）** | 高度な絞り込み検索・お気に入り・会員機能・CRM連携 | **中〜高** |
| MVP | 物件一覧＋詳細のみ・最速稼働・段階拡張 | 低・最速 |

**フェーズ提案（推奨）：**
Ph.1 → MVP早期リリース（物件一覧・詳細・問い合わせ）
Ph.2 → 検索機能強化・会員化・CRM連携

### コンペ戦略

- 予算・企画確定済みなので「動くものを早く出す」MVP戦略が刺さる
- KurocoをすでにWebで認識している（指名相談）→ 製品採用意向が高い
- 他社との差別化：APIファースト設計で「将来の拡張が容易」を強調

## 4. 技術的な合意事項

### アーキテクチャ概要

```
[管理者] 
  → Kuroco CMS管理画面 で物件情報を登録・編集・公開

[Kuroco CMS（SaaS）]
  ├── REST/GraphQL API → Next.jsフロントエンド
  ├── Webhook → CRM/MAツール（問い合わせリード送信）
  └── 会員認証 → マイページ・お気に入り管理

[Next.jsフロントエンド]
  ├── 物件検索・絞り込みUI（F-01）
  ├── 物件一覧・詳細ページ
  ├── お気に入り機能（F-02）
  ├── お問い合わせフォーム（F-03）
  └── マイページ（会員）

[ユーザー（起業家・企業担当者）]
  → ブラウザ / スマートフォンでアクセス
```

### 主要フロー設計

| フロー | 設計方針 |
|--------|---------|
| 物件登録フロー | Kuroco管理画面でコンテンツ型（物件）を定義→登録→公開 |
| 検索フロー | Kuroco APIのフィルタ機能＋B-01拡張で複合条件検索を実現 |
| 問い合わせフロー | フォーム送信→Kurocoワークフロー→担当者メール→CRM Webhook |
| 会員認証フロー | Kuroco標準の会員認証API→Next.js側でセッション管理 |
