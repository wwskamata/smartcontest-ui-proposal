## スキル①｜提案・モックアップ作成（wws-proposal）

**発動トリガー**：「モックアップ」「画面フロー」「社内総合シート」「キャンペーン提案」「プロダクト選定」

### ワークフロー

**STEP 1｜相談内容の構造化**

相談を受けたら以下の軸で整理する：

| 項目 | 内容 |
|------|------|
| 依頼者 | エンド / 代理店 / 自社　・業種 |
| 目的 | 何を達成したいか |
| 対象ユーザー | 誰が参加するか・認証方式の制約 |
| キャンペーン形式 | スタンプ・レシート・SNS・クーポン等 |
| 課題 | 何が難しいか・現場制約 |
| 制約 | 技術・運用・予算・納期 |

整理後、確認事項があれば1つだけ質問してから次のSTEPへ。

**STEP 2｜プロダクト選定**

| 要件 | 推奨プロダクト |
|-----|--------------|
| 同一場所に繰り返し来訪・ポイント累積 | SmartStampCard（マイレージOP） |
| スポットを1回ずつ巡るラリー | SmartStampRally |
| レシート・領収書で購買証明 | SmartRecieco |
| GPS判定＋レシート判定を統合 | Card＋Recieco（カスタム連携） |
| SNS/UGCキャンペーン | SmartPRシリーズ |

**プラン提示の型（3段階）：**
1. **標準プラン** — 既存機能のみ・最短納期・低コスト
2. **推奨プラン** — 一部カスタマイズ・バランス型（コンペ推奨）
3. **フルカスタムプラン** — 完全要件対応・高コスト

**STEP 3｜ユーザーフロー設計**

フロー設計前に確認すること：
- 認証方式（LINEログイン / メールアドレス / SNS）
- 付与タイミング（即時 / 審査後）
- 特典形式（後日抽選 / インスタントウィン / クーポン）
- 事務局オペレーション頻度（日次 / リアルタイム / 無人）

**STEP 4｜モックアップの記述ルール**

HTMLモックアップで使う画面バッジ：
- `F-xx`（赤）= フロントエンド カスタム開発
- `B-xx`（赤）= バックエンド カスタム開発
- `標準`（緑）= 既存機能そのまま
- `Receico`（青）= SmartReciecoプロダクト
- `B-xx設定`（黄）= 設定作業のみ（コーディング不要）

画面フローの構成：
- 左カラム（320px固定）: 画面フロー縦1列
- 右カラム（残り幅）: 工数算定依頼テーブル

**STEP 5｜工数算定テーブルの列**

`ID` / `区分（フロント/バック/設定）` / `機能名` / `要件概要` / `難易度（小/中/大）` / `確認事項`

**重要ルール：**
- OCR結果はユーザー画面に表示しない（管理側のみ）
- Card＋Reciecoの統合はカスタム開発が必要（標準では連携不可）
- 外向け資料では「カスタム開発」→「拡張機能」「標準機能＋対応」に言い換える

## スキル②｜ピッチデッキ作成（wws-pitch-deck）

**発動トリガー**：「提案資料」「パワポ」「クライアントに見せるスライド」「ピッチデッキ」

### スライド構成（固定・7〜9ページ）

| ページ | 内容 |
|--------|------|
| P1 | 表紙（キャンペーン名・クライアント名・プロダクトバッジ・日付） |
| P2 | 課題・背景（クライアントが抱える問題を3〜4枚のカードで） |
| P3 | 提案概要（3段階プラン比較・推奨プランを強調） |
| P4 | システム構成（ユーザー→プロダクト→事務局の流れ） |
| P5 | 費用感・スケジュール（テーブル＋4フェーズガント） |
| P6 | ユーザーフロー図（横長・全ルートを1枚に） |
| P7〜9 | 画面フロー（1ページ=最大4画面・カスタム画面を優先） |

### デザイン原則

| 用途 | カラー |
|------|--------|
| アクセント（オレンジ） | `#E87722` |
| グリーン | `#4CAF50` |
| カスタム背景 | `#FEE2E2` / テキスト `#991B1B` |
| 標準背景 | `#DCFCE7` / テキスト `#166534` |
| MapPenguin | `#E8F5E9` / テキスト `#1B5E20` |

- 1スライドにつき1メッセージ
- 箇条書きは最大5項目・1行1項目
- アイコン・図形でテキスト量を絞る

### 情報収集（STEP1）

以下がわかっていれば即生成。なければ1問だけ確認：
- クライアント名・業種・目的
- 使用プロダクト
- 費用感（概算）・スケジュール感
- カスタマイズの有無

### 画面フローの画面選定優先順位

1. カスタムバッジ（F-xx / B-xx）がついている画面 → 必ず含める
2. フロー上の起点・分岐点（ログイン・マイページ）
3. 標準UIのみの画面 → 省略可

## スキル③｜HTML公開（wws-publish）

**発動トリガー**：「GitHubに上げて」「パスワードかけて公開」「staticrypt」「URLを共有したい」

### 手順

```
1. staticrypt で暗号化
   npx staticrypt@latest {input.html} -p {PASSWORD} --short
   --template-title "WWS 提案資料"
   --template-instructions "担当者よりお伝えしたパスワードを入力してください"
   --template-color-primary "#e87722"
   --remember 7
   -d encrypted_output/

2. GitHub リポジトリ作成 & push
   gh repo create {repo-name} --public --source=. --remote=origin --push

3. GitHub Pages 有効化
   gh api repos/wwskamata/{repo-name}/pages
   --method POST -f "source[branch]=master" -f "source[path]=/"

4. URL確認
   https://wwskamata.github.io/{repo-name}/
```

### リポジトリ命名規則
```
{案件キーワード}-{プラン}
例: transit-planb, transit-plana, entre-support-proposal
```

### セキュリティ原則
- パスワードはメール本文に書かない（Slack DM または電話で別途共有）
- 公開停止: `gh repo delete {name} --yes`
- 管理台帳: `publish_log.md` に記録

## スキル④｜プレスリリース作成（wws-prtimes）

**発動トリガー**：「プレスリリース」「PR TIMES」「リリース文」「告知文」「メディア向け」

### 必須ルール

- **タイトル：40字以内**（全角・半角・記号・スペース含めすべて1文字）
- **リード文：200字以内**
- タイトルは必ず自己検証して `（XX字 ✅）` を付記する

### 標準構成

```
## タイトル案（3案提示）
## リード文（200字以内）
## ▼ 背景：[課題]
## ▼ [製品名] とは
## ▼ 主な特長
## ▼ 導入事例
## ▼ 詳細・お申込み
## ■ 会社概要
## ■ お問い合わせ先
```

### タイトルの推奨パターン
```
[製品名]が[対象]向けに[特典/成果]。[期限/条件]。
```
例：`工場DXをゼロコストで。MapPenguin Pro、7月末まで無料。`（28字 ✅）

### リード文の型
- 第1文：主語＋発表内容
- 第2文：なぜ今か・背景
- 第3文：主な特典・数値・読者へのベネフィット

**発動トリガー**：「Kuroco」「Admin-DX CMS」「ヘッドレスCMS」「CMS選定」「Next.js＋CMS」

## 共通ルール（全スキル共通）

### 提案時の基本スタンス
- 「要件を100%満たせなくても、目的を達成できる提案をする」
- コンペ案件では「スケジュール確実性」と「費用の透明性」を前面に出す
- フェーズ提案（Ph.1/Ph.2）で予算・スケジュール感に合わせる

### バッジ・表記ルール（外向け資料）
| 内部表記 | 外向け表記 |
|---------|----------|
| カスタム開発 | 拡張機能 / 標準機能＋対応 |
| SE算定 | 開発工数確認中 |
| B-01（内部ID） | 記載しない |

### 確認待ち事項を出す場合
表形式で `項目 / 影響範囲 / 確認先` の3列で出す。
